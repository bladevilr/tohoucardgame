# 东方大巴扎 V2：核心系统设计稿
> 基于 The Bazaar 反推构筑法 + 上菜顺序创新 + 共享评委状态博弈

---

## 一、对决模型：上菜顺序制

### 1.1 与旧模型的本质区别

| | V1 (Tick-DPS) | V2 (上菜顺序) |
|---|---|---|
| 时间模型 | 30s 连续 Tick，所有菜同时跑 CD | 离散回合，菜品按玩家排列的顺序逐道上 |
| 核心体验 | 看它们打（自走棋） | 我编排了这一切（套路设计） |
| 速度含义 | CD 减少 = DPS 提高 | 烹饪时间短 = 我的序列推进更快 = 先手优势 |
| 位置含义 | 棋盘格子（空间邻接） | 队列位置（时间先后） |
| 对抗方式 | 分数比大小 | **共享评委状态** = 我上的菜影响你菜的得分 |

### 1.2 执行流程

```
准备阶段：
  玩家A 排列出菜顺序: [前菜1] → [汤] → [主菜1] → [主菜2] → [甜点]
  玩家B 排列出菜顺序: [前菜1] → [主菜1] → [主菜2] → [台] → [收尾]

对决阶段（双方并行推进）：
  时间轴 ─────────────────────────────────────────────→
  
  玩家A:  [前菜1 烹饪2s] → 上菜! → [汤 烹饪1.5s] → 上菜! → [主菜 烹饪4s] → ...
  玩家B:  [前菜1 烹饪3s] ───────→ 上菜! → [主菜1 烹饪3s] ──────→ 上菜! → ...
  
  评委状态: ──────────── 共享，双方的菜都在影响 ────────────
```

**关键规则：**
1. 每次只能有一道菜在烹饪（前台），烹饪时间 = 菜品的 `cooldown` 属性
2. 烹饪完成 = 自动上菜 = 触发效果 + 计分 + 修改评委状态
3. 上完立刻开始烹饪下一道
4. 双方各自推进，**速度快的一方自然多上几道菜**
5. 对决在固定时间（如 30s）后结束，或双方都上完所有菜时结束

### 1.3 后台炖煮（双轨制）

某些菜品带有 `[slow_cook]` 标签（炖煮/发酵/低温慢煮）：
- 可以在对决开始时放入"后台炉灶"
- 在后台持续计时（不占用前台出菜序列）
- 到时间后**自动上菜**（插入到当前前台序列的间隙）
- 后台菜品的烹饪时间通常较长（6-10s），但属性很高
- 后台槽位有限（1-2个，可通过厨具扩展）

**策略价值**：你是选择全部前台速攻（多上菜但单道弱），还是分1-2道去后台慢炖（前台少上一道但后台菜是核弹）？

### 1.4 胜负判定

对决结束时：
- 计算双方总分
- 分差决定声望伤害（沿用现有公式）
- PvE 失败不扣声望

## 二、评委状态机（共享战场）

评委不是计分器，是一个**有生理和心理反应的角色**。双方的菜都作用于同一位评委。

### 2.1 状态轴

| 状态 | 范围 | 增加方式 | 减少方式 | 对得分的影响 |
|------|------|---------|---------|------------|
| **饱腹度** | 0-100 | 每道菜 +size×8 | 无（只增不减） | >60: 后续菜得分×0.8; >80: ×0.5; =100: 拒食 |
| **味觉阈值** | 0-10 per味型 | 吃同类口味菜 | 吃"清口"菜归零 | 同味型菜：阈值越高得分越低（递减） |
| **惊喜度** | 0-10 | 吃到新菜系/新标签 | 吃到重复的 | 高惊喜=额外评分奖励 |
| **期待值** | 1-10 | 吃到高卖相/高概念菜 | 时间衰减 | 后续菜低于期待值=扣分 |
| **心情** | -5~+5 | 吃到对口菜 | 吃到厌恶菜 | 乘法修正，正=加分负=扣分 |

### 2.2 菜品对状态的影响（自然因果，非抽象 Bonus）

菜品**不写**"+15% CD"这种抽象描述。而是写自然语言效果，系统翻译为状态变化：

```
麻婆豆腐:
  serve_effect:
    satiety: +10          # 小菜，不太占肚子
    taste_threshold:
      spicy: +3           # 吃完辣度阈值拉高
    mood: +1 if judge.likes("spicy") else -1  # 喜辣+1，不喜辣-1
    
凉拌黄瓜:
  serve_effect:
    satiety: +5           # 几乎不占肚子  
    taste_threshold:
      spicy: -3           # 解辣，降低辣度阈值
      ALL: -1             # 清口，所有味型阈值-1
    surprise: +1 if prev_dish.has_tag("spicy")  # 辣后来清淡=惊喜
    
红烧狮子头:
  serve_effect:
    satiety: +25          # 大菜，很占肚子
    taste_threshold:
      rich: +2            # 浓厚口味
    expectation: +2       # 大菜抬高期待
    mood: +2 if judge.likes("homestyle")  # 家常菜爱好者狂喜
```

### 2.3 得分公式

```
单菜得分 = 基础风味 
         × 味觉新鲜度修正（基于该味型阈值的递减函数）
         × 饱腹度修正（饱了打折）
         × 惊喜度加成（有惊喜额外乘）
         × 期待值修正（低于期待扣分，超出加分）
         × 评委心情修正
         × 评委个人偏好乘区
         × 厨具/技能乘区
```

**核心博弈点**：所有"修正"都是双方共享的评委状态决定的。我先上辣菜 → 评委辣度阈值升高 → 你后上的辣菜打折。

### 2.4 评委个性（抽象数值的归属地）

| 评委 | 偏好标签 | 特殊机制 |
|------|---------|---------|
| **四季映姬** | 无偏好（公正） | 惊喜度权重×2，期待值衰减极慢，打分极严 |
| **射命丸文** | 新奇、fusion | 重复菜品惩罚×2，新菜系奖励×2，每道菜都想要不一样 |
| **八云紫** | 大菜、高概念 | 饱腹度上限+30（能吃），忽略前3道小菜 |
| **西行寺幽幽子** | 全部（饕餮） | 饱腹度永远不满，但味觉阈值上升极快 |
| **蓬莱山辉夜** | 稀有、精致 | 铜级菜扣心情，金钻级菜+心情 |

## 三、三层架构

### 第一层：菜品 = 自然因果
- 辣的就是辣的，浓的就是浓的
- 效果是"对评委做了什么"
- 玩家靠生活常识和发现来构筑

### 第二层：评委 = 偏好滤镜（抽象加成在这）
- 不同评委对不同菜系、标签有天然好感/厌恶
- 偏好以"心情变化量"和"分数乘区"体现
- 每局随机评委 → 同一套菜面对不同评委要调整顺序

### 第三层：厨具/角色技能 = 玩家侧加速器
- 中华炒锅 → 炒菜的烹饪时间-0.5s（理由："锅好"）
- 保温箱 → 后台菜品不会因为时间太长品质衰减
- 角色被动仍保留，但效果从抽象数字改为叙事化描述

## 四、菜品设计原则（多维"端口"）

每道菜至少横跨 3 个维度。菜品不属于"流派"——流派是玩家自己从维度组合中**发现**的。

### 4.1 维度列表

1. **菜系**：中/和/洋/夜/甜/药（6种）— 评委偏好、同系联动
2. **口味标签**：辛辣/鲜美/清淡/浓厚/酸甜/苦… — 味觉阈值博弈
3. **烹饪法**：烤/炸/蒸/煮/炒/生/炖… — 烹饪时间相关、同法联动
4. **体量**：小(1)/中(2)/大(3) — 饱腹度影响、速度策略
5. **顺序效果**：对前/后道菜的条件触发 — 组合链
6. **评委状态操控**：拉高阈值/清口/涨期待/涨饱腹 — 干扰与自利

### 4.2 示例：一道菜的多端口设计

```
担担面 (dan_dan_noodles):
  cuisine: "chuuka"                    → 菜系端口
  tags: ["spicy", "noodle", "stir_fried", "small"]  → 口味+烹饪+体量 端口
  size: 1, cooldown: 2.0              → 速攻端口
  base_stats: {flavor:7, presentation:2, technique:4, aroma:6}
  
  serve_effect:                        → 评委状态端口
    satiety: +8
    taste_threshold: {spicy: +2}
    
  sequence_triggers:                   → 顺序端口
    - when: "prev_dish_has_tag"
      tag: "spicy"
      effect: {flavor_bonus: +3}       # "辣上加辣"连击
      
    - when: "next_dish_has_tag"  
      tag: "light"
      effect: {buff_next: {flavor_mult: 1.3}}  # "辣后清淡"反差加成给下道菜
```

这道菜可以：
- 放在辛辣序列里连续输出（辣上加辣）
- 放在清口菜前面当铺垫（辣后清淡给下道菜加分）
- 凑中餐维度
- 凑炒菜维度
- 凑小菜军团
- 抬高评委辣度阈值压制对手的辛辣菜

**食材的作用**：花椒拖上去 → 加了 `[numbing]` 标签和 `spicy: +1` → 端口连接改变

## 五、经济与难度曲线

### 5.1 日结构（参考 The Bazaar）

| 时段 | 类型 | 内容 | 设计意图 |
|------|------|------|---------|
| 行动1 | 三选一 | 商贩/事件/物品 | 开局定方向 |
| 行动2 | 三选一 | 商贩/事件/物品 | 补强/转向 |
| 行动3 | PvE | 打AI对手 | 中期检验 |
| 行动4 | 三选一 | 商贩/事件/物品 | 最后调整 |
| 行动5 | PvP | 异步对战 | 结算日 |

每日 **4次三选一 + 1次PvE + 1次PvP**。

对比原版的6行动结构，减去1个事件、把PvE提前到行动3，节奏更紧凑。

### 5.2 资源曲线

| Day | 基础收入 | 期望总金币 | 可用品阶 | 期望板面实力 |
|-----|---------|-----------|---------|------------|
| 1 | 10(起始) | 10 | 铜 | 3-4道铜菜，无厨具 |
| 2 | 6+1=7 | 12-15 | 铜+银 | 5-6道菜，1厨具 |
| 3 | 6+2=8 | 14-18 | 铜银+少量金 | 6-7道菜，可能有2星 |
| 5 | 6+4=10 | 18-24 | 银金为主 | 7-8道菜，套路成型 |
| 7 | 6+6=12 | 22-30 | 金+钻出现 | 8-9道菜，核心Combo就位 |
| 10 | 12(封顶) | 25-35 | 金钻为主 | 满编，3星核心，终极形态 |

### 5.3 声望与淘汰节奏

- 起始 20 声望
- PvP败：-2 基础 - floor(分差/80)
- 连败补偿：3连败+2金, 5连败+3金
- **第一次归零**：声望锁1，额外三选一加强机会（保底机制）
- **第二次归零**：淘汰

**设计意图**：前3天允许试错（声望够扛2-3次败），Day 4-7 是分水岭（套路必须成型），Day 8+ 是决战期。

### 5.4 PvE 难度曲线

| Day | PvE对手实力 | 奖励 | 失败惩罚 |
|-----|-----------|------|---------|
| 1 | 3道铜菜，无套路 | 3金 | 无 |
| 2 | 4道铜银菜 | 4金 | 无 |
| 3 | 5道银菜+1厨具 | 5金+随机食材 | 扣2金 |
| 5 | 7道银金+套路雏形 | 6金+选择奖励 | 扣3金 |
| 7 | 8道金菜+完整套路 | 8金+稀有物品 | 扣4金 |
| 10+ | 满编金钻+优化序列 | 10金+钻石物品 | 扣5金 |

## 六、自动化平衡测试框架

### 6.1 扩展 HeadlessSimulator

现有 `HeadlessSimulator` 需要适配新的上菜顺序模型：

```
输入:
  loadout = {
    chef_id: "meiling",
    serve_queue: ["mapo_tofu", "cucumber_salad", "lions_head", ...],  # 有序！
    back_burner: ["dongpo_pork"],  # 后台炖煮
    tool_ids: ["chinese_wok"],
    ingredient_applied: {"mapo_tofu": ["sichuan_pepper"]},
  }

输出:
  {
    winner: 0,
    scores: [850, 720],
    serve_timeline: [
      {time: 0.0, player: 0, dish: "mapo_tofu", score: 45, judge_state: {...}},
      {time: 2.0, player: 0, dish: "cucumber_salad", score: 62, judge_state: {...}},
      {time: 3.0, player: 1, dish: "dish_a", score: 38, judge_state: {...}},
      ...
    ],
    judge_final_state: {...},
  }
```

### 6.2 自动化测试流程

```
Phase 1: 单 Build 强度测试
  - 枚举所有可能的 Build 方向（中餐纯、辛辣混、速攻等）
  - 每个 Build 用贪心 AI 自动选卡/排序
  - 对打随机 Build 1000 次
  - 输出：每个 Build 的平均得分、胜率、方差

Phase 2: Build 对 Build 矩阵
  - N个主流Build两两对打，各1000场
  - 输出：N×N 胜率矩阵
  - 平衡目标：任何Build胜率在 45%-55% 之间
  - 报红：某 Build 对所有其他 Build 胜率 > 55% → 需削弱

Phase 3: 经济曲线验证
  - 模拟完整15天游戏流程（含商店/三选一/PvE/PvP）
  - 跑10000局
  - 检查：
    - Day 3前淘汰率应 < 5%（新手保护期）
    - Day 7平均存活率 60-70%
    - Day 10+平均存活率 30-40%
    - 最终赢家的平均天数 8-12

Phase 4: 迭代调参
  - 参数空间：菜品base_stats、烹饪时间、评委状态系数
  - 自动扫描关键参数±20%范围
  - 找到使所有Build胜率最接近50%的参数组合
  - 输出推荐参数表
```

### 6.3 具体脚本架构

```
tools/
├── HeadlessSimulator.gd    # 单场模拟（需重写适配V2）
├── BuildGenerator.gd       # 自动生成各种Build方向的loadout
├── BalanceMatrix.gd        # Build×Build 胜率矩阵
├── EconomySimulator.gd     # 15天完整流程模拟
├── AutoTuner.gd            # 自动调参（参数扫描+优化）
└── balance_report.md       # 输出报告（自动生成）
```

## 七、商贩系统（三选一）

### 7.1 商贩 = 维度组合

商贩不按品类分，按**维度交叉**分。每个商贩的物品池覆盖 2-3 个维度：

- "夜雀屋台"：夜市×烤制×小型 → 速攻/焦香方向
- "红魔馆采办"：西餐×精致×大型 → 卖相/期待值方向
- "永远亭药铺"：药膳×清淡×食材 → 环境控制/解毒方向
- "魔法森林"：随机×高方差 → 赌博/转向
- "角色来访"：其他角色的专属物品 → 跨角色组合

### 7.2 三选一生成逻辑

每次三选一从以下类型中随机抽3个不重复的：
- 商贩 ×（各种维度组合）：70% 权重
- 奇遇事件（直接获得效果）：20% 权重
- 特殊事件（赌博/升级/交换）：10% 权重

### 7.3 气泡 UI 信息透明

每个气泡明确显示：
- 商贩立绘 + 名字
- 涉及维度标签（如 `[辛辣] [小型] [中餐]`）
- 1-2个代表性物品预览
- 奇遇事件直接显示结果（"获得 8 金币"）
